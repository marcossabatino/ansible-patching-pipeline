Production-grade **patch management pipeline** for Ubuntu EC2 instances, built on Ansible. The pipeline implements the full operational cycle вҖ” pre-check, patch, conditional reboot, post-check, and report вҖ” with safe pre-conditions, environment-aware execution, and rich TXT + HTML evidence of completion.

---

## Overview

The playbook applies the same pipeline to every managed host, one at a time (`serial: 1`), so a single bad patch never takes down the entire fleet simultaneously.

| Stage | What happens | Abort if |
|---|---|---|
| **PRE-CHECK**  | Verifies disk, memory, and critical services | Below thresholds or service down |
| **SNAPSHOT**   | Captures installed packages + pending upgrades | Never aborts |
| **PATCH**      | Runs `apt` upgrade (`safe` / `full` / `security`) | `apt` error |
| **REBOOT**     | Conditional reboot when `/var/run/reboot-required` exists | Never aborts (reboot timeout вҶ’ task fails) |
| **POST-CHECK** | Re-verifies services, computes package diff | Any critical service down after patch |
| **REPORT**     | Generates TXT + HTML and fetches them to control node | Never aborts |

## Enterprise Context

This project is structured so it can be lifted directly into **Ansible Automation Platform (AAP) / Ansible Tower**:

- **Job Template** вҶ’ `playbooks/patch.yml`
- **Survey fields** map one-to-one to variables (`patch_upgrade_type`, `reboot_if_required`, `pinned_packages`).
- **`serial: 1`** gives a rolling patch вҖ” a production requirement when the fleet serves live traffic. One host at a time preserves capacity and lets automation roll forward through partial failures.
- **Environment-based inventory groups** (`sandbox`, `production`) consume `group_vars/<env>.yml`, so the same role applies different patching aggressiveness by environment вҖ” sandbox gets `full`, production gets `security` only.
- **Reporting** вҖ” the generated HTML report is designed to be attached to a **ServiceNow change ticket** or uploaded to an **S3 bucket** via an AAP post-job workflow step, producing an audit trail without manual screenshots.
- **Compliance** вҖ” the TXT report includes before/after package lists, reboot status, and service verification вҖ” sufficient evidence for typical change-management records.

## Terraform Reuse

The `terraform/` module is **adapted** from the companion project [`ansible-linux-hardening`](https://github.com/marcossabatino/ansible-linux-hardening) to keep infrastructure portable across portfolio projects. Differences:

- `project_name` default вҶ’ `patching-pipeline` (no "lab" designation)
- New `environment` variable (default `sandbox`) вҖ” drives patch aggressiveness
- Control node `user_data` also installs `boto3` + `botocore` for AWS dynamic inventory
- Resource identifiers renamed from `cis_lab` вҶ’ `patching`
- SSH key paths default to `~/.ssh/patching-pipeline` (project-scoped)

## Step-by-Step Execution

### 1. Clone

```bash
git clone git@github.com:marcossabatino/ansible-patching-pipeline.git
cd ansible-patching-pipeline
```

### 2. Generate an SSH key (if you do not already have one)

```bash
ssh-keygen -t ed25519 -f ~/.ssh/patching-pipeline -N ""
```

### 3. Provision the infrastructure with Terraform

```bash
cd terraform
terraform init
terraform apply -auto-approve
cd ..
```

Expected output ends with two managed nodes, one control node, and `inventory/hosts.ini` written.

### 4. Verify connectivity

```bash
ansible managed_nodes -m ping
```

Expected: `SUCCESS` on every host.

### 5. Run pre-check only (safe, no changes)

```bash
ansible-playbook playbooks/patch.yml --tags pre_check
```

Expected: all `assert` tasks report `PASSED`.

### 6. Run the full pipeline

```bash
ansible-playbook playbooks/patch.yml
```

Expected final output (per host):

## Tag Reference

| Tag | Stage triggered |
|---|---|
| `pre_check` / `check` | Pre-check only (safe, no changes) |
| `snapshot`            | Package list capture only |
| `patch` / `upgrade`   | Patching only (skip pre-check) |
| `reboot`              | Reboot decision + reboot |
| `post_check`          | Post-check only |
| `report`              | Report generation only (also runs with `always`) |

## Customization via Variables

| Variable | Default | Example override |
|---|---|---|
| `patch_upgrade_type`    | `safe` | `-e "patch_upgrade_type=security"` |
| `reboot_if_required`    | `true` | `-e "reboot_if_required=false"` |
| `pinned_packages`       | `[]`   | `-e '{"pinned_packages": ["nginx"]}'` |
| `min_disk_free_percent` | `20`   | `-e "min_disk_free_percent=30"` |
| `min_memory_free_mb`    | `256`  | `-e "min_memory_free_mb=512"` |
| `serial` (playbook)     | `1`    | Edit `playbooks/patch.yml`: `serial: 2` |

## Validation Tests

| Test | Command | Pass criteria |
|---|---|---|
| Pre-check passes        | `ansible-playbook playbooks/patch.yml --tags pre_check` | exit 0 |
| Disk usage query        | `ansible managed_nodes -m shell -a "df / --output=pcent \| tail -1"` | under threshold |
| Services running        | `ansible managed_nodes -m shell -a "systemctl is-active ssh rsyslog"` | `active` |
| Reboot required         | `ansible managed_nodes -m stat -a "path=/var/run/reboot-required"` | reports `exists` correctly |
| Reports generated       | `ls reports/patch_report_*.txt reports/patch_report_*.html` | both present |
| Idempotency             | Re-run `ansible-playbook playbooks/patch.yml` | `changed=0` on `apt` upgrade |
| Lint                    | `ansible-lint playbooks/patch.yml` | 0 warnings |
